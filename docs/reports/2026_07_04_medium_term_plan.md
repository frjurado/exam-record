## Step 1 — Migrate Models to SQLAlchemy 2.0 Style (`Mapped`/`mapped_column`)

### Why
`app/models.py` still uses the SQLAlchemy 1.x `Column(...)` API. The 2.0-style `Mapped[T]` / `mapped_column(...)` API gives mypy real type information for ORM attributes (currently mypy can't catch `user.email` being used as an `int`, for instance), enables IDE autocomplete, and is the direction SQLAlchemy itself is moving — new features increasingly assume it. This is worth doing now, before more services are built against the current model shape, since it's a mechanical but invasive change (every model, every column).

It's also the natural place to fix two smaller items flagged in the same file: the missing `back_populates` on `Vote.user` (§ 3.4) and the missing indexes on foreign key columns used in frequent filters/joins (§ 9.2) — `reports.event_id`, `reports.user_id`, `votes.report_id`, `votes.user_id`.

### How
- Convert every class in `app/models.py` from `Column(...)` to `id: Mapped[int] = mapped_column(primary_key=True, index=True)` style, preserving nullability, defaults, uniqueness, and FK targets exactly.
- Add `Vote.user = relationship("User", back_populates="votes")` and the matching reverse `votes = relationship("Vote", back_populates="user")` on `User`.
- Add `index=True` to `Report.user_id`, `Report.event_id`, `Vote.report_id`, `Vote.user_id`.
- Generate an Alembic migration for the new indexes (`alembic revision --autogenerate -m "add fk indexes"`) — review the autogenerated migration carefully since SQLite index creation via Alembic sometimes needs `batch_alter_table`.
- Run `pytest` and `mypy app` after.

### Decisions to make
- **Do this in one PR or split model conversion from the index migration?** Recommend one PR — they touch the same file and the index migration is trivial once the models are converted, splitting just adds review overhead for no isolation benefit.
- **Should `relationship()` calls also gain explicit `Mapped[list["Report"]]` type annotations?** Recommended yes, since that's the whole point of the 2.0 style and half-converting (columns typed, relationships untyped) would leave mypy blind on the more error-prone half.

**Mode:** `--plan` — touches every model and needs a migration; review before executing.

**Review:** `pytest`, `mypy app`, and `alembic upgrade head` against a fresh copy of the dev DB to confirm the migration applies cleanly.

## Step 2 — Rate Limiting on Critical Endpoints

### Why
There's currently no protection against brute-forcing `/api/auth/login` (magic-link request spam), report spam, or vote manipulation via scripted requests. This is a public-facing app taking anonymous submissions, so it's a real abuse vector, not a theoretical one.

### How
- Add `slowapi` as a dependency.
- Add a `Limiter` keyed on remote address in `app/main.py`, wired into the FastAPI app via `app.state.limiter` and the standard exception handler.
- Apply limits to: magic-link request endpoint (e.g. `5/minute` per IP), report creation (e.g. `10/hour` per IP), and voting (e.g. `20/hour` per IP).
- Add a test asserting the 429 response after exceeding a limit (slowapi supports overriding the limiter's storage/window for tests).

### Decisions to make
- **In-memory vs. Redis-backed limiter storage.** In-memory is fine for the current single-instance Fly.io deployment; only revisit if the app scales to multiple machines (shared state would be needed then). Recommend starting in-memory — no need to add Redis just for this.
- **Limit granularity: per-IP only, or per-IP + per-user for authenticated actions?** Recommend per-IP only for now — simpler, and the abuse cases in scope (anonymous spam, brute force) are IP-shaped. Per-user limits can be layered on later if abuse patterns show it's needed.

**Mode:** Direct execution — additive, limits are conservative and easy to tune after observing real traffic.

**Review:** Manually hit an endpoint past its limit locally and confirm a 429; run `pytest`.

## Step 3 — Caching Layer for Frequently Accessed Data

**Status: ✅ COMPLETED (2026-07-04)** — implemented as `app/services/reference_data_service.py`, with one correction to the premise below: the landing page turned out to be static HTML with no "list all regions/disciplines" query anywhere. What actually repeats on every discipline/exam page view is a single-row Region/Discipline lookup *by slug* (in `ExamService.get_discipline_context` and `main.py`'s `contribute_page`), so those got cached instead — same static-data intent, matching what the code actually does. Cached values are plain `id`/`slug`/`name` dataclasses rather than live ORM instances, so a cached entry can never end up bound to a closed session. `get_exam_context`'s combined Region+Discipline+ExamEvent join query was left untouched (already one query, not a separate cacheable lookup), and lookup misses (`None`) are never cached, so a newly-added region is visible immediately. Verified end-to-end against the real dev DB: the SQL echo log confirmed the Region/Discipline `SELECT`s fire on the first request and are skipped entirely on a repeat request, while the `ExamEvent`/votes queries still run every time as intended.

### Why
`Region` and `Discipline` lists are read on nearly every page load (they populate nav/dropdowns) and change essentially never. Querying them from SQLite on every request is wasted work, and this is the first genuinely low-risk performance win available — no query logic changes, just avoiding redundant reads of static reference data.

### How
- Add a simple in-process cache (`functools.lru_cache` won't work directly on an async function — use a small module-level cache with manual invalidation, or `asyncache`/`cachetools.TTLCache`) for `get_all_regions()` and `get_all_disciplines()`-style lookups.
- Cache with a TTL (e.g. 1 hour) rather than forever, so a manual DB edit to seed data doesn't require a redeploy to take effect.
- Do **not** introduce Redis for this — the dataset is tiny (a handful of rows) and the app is single-instance; Redis would add an operational dependency for no real benefit at this scale.

### Decisions to make
- **What's actually worth caching?** Recommend scoping this to `Region` and `Discipline` only (truly static reference data). Do not cache `ExamEvent`/`Report`/vote aggregates — those change with every user submission and caching them risks showing stale consensus data, which undermines the whole point of the app.

**Mode:** Direct execution — small, contained, easy to revert if it causes staleness complaints.

**Review:** Confirm nav/dropdowns still reflect DB state after adding a region manually (wait out the TTL or restart), and that `pytest` still passes.

## Step 5 — Monitoring (Error Tracking)

### Why
Right now, production errors are only visible via Fly.io logs (or `error.log`), which means issues surface only when a user reports them or someone happens to be watching logs. For a side project with unpredictable traffic, an error tracker (Sentry) that pages/emails on new exception types is a much lower-effort signal than log-watching.

### How
- Add `sentry-sdk[fastapi]`, initialize it in `app/main.py` guarded by `settings.ENVIRONMENT == "production"` and a `SENTRY_DSN` env var (optional, like the existing `RESEND_API_KEY`/Turnstile pattern — omit to disable).
- Do not add a second tool (DataDog) alongside it — Sentry alone covers the actual need here (uncaught exceptions), and running two observability vendors for a project this size is overhead without benefit.

### Decisions to make
- **Sentry vs. a self-hosted/free alternative (e.g. GlitchTip)?** Recommend Sentry's free tier — it's the path of least setup effort, and this app's error volume won't approach the free tier's limits. Revisit only if cost becomes a factor.

**Mode:** Direct execution — additive, disabled by default via missing env var, same pattern as existing optional integrations.

**Review:** Trigger a deliberate exception locally with `SENTRY_DSN` set to a test project and confirm it appears in the Sentry dashboard.
